# Program Teacher Demo Handoff

This handoff is for the next staff/admin/teacher click-around using the local demo seed. It assumes Prompt 1 and Prompt 2 hardening have been applied, validated, committed, and pushed.

`C:\SeniorProjectApp1.0\.secrets\demo-staff-logins-*.json`

| Persona | Fake email | Role | Scope | Why this one |

| Primary IT Program Teacher | `teacher-it-01@demo-staff.capstone.test` | Program Teacher | IT program | Shows scoped program dashboard, missing mentor/evidence filters, review queue, evidence, and student detail. |

## Preflight

1. Confirm local demo data exists or reseed safely:
   `npm run seed:demo:local:dry-run`
2. When a reset is intended for local fake data only:
   `npm run seed:demo:local:reset`
3. Prove the local demo data and role-scoped routes:
   `npm run prove:demo:local`
4. Run the app locally:
   `npm run dev:alpha`
5. Open:
   `http://127.0.0.1:8788/workspace`

The seed is local-only, deterministic, `.test`-domain-only, and protected by repo/branch/admin checks. It does not create student passwords and does not write production data.

## Click Path

1. Sign in as `teacher-it-01@demo-staff.capstone.test`.
2. Confirm the nav shows Program Teacher sections: Overview, Students, Program Dashboard, Review Queue, Mentor Assignments, Operations, Presentation, and Account.
3. Confirm the nav does not show Global Admin tools, Users & Access, Audit, or Archive / Exports.
4. Open Program Dashboard.
5. Review the metric cards: Total Students, On Track, Behind / Needs Support, Missing Evidence, Needs Review, Missing Mentor, and Needs Revision.
6. Click Missing Mentor, then confirm the Student Directory opens with a missing-mentor filter and only IT students for the selected site.
7. Open a student detail row. The detail panel should open at the top and show header, progress, mentor status, evidence/review context, and next action.
8. Return to Program Dashboard.
9. Click Missing Evidence, then confirm the Student Directory opens with `evidenceStatus=missing`.
10. Return to Program Dashboard.
11. Click Needs Review, then confirm the Review Queue opens with submitted work.
12. Open a submitted/revision row's student detail or review history. The view should stay scoped to the teacher's program and show evidence/review context.
13. Search for `Revision Loop Demo`, clear filters, and confirm the directory returns to the scoped IT roster.

## Expected Demo Signals

The proof script asserts the Program Teacher path has:

- a known fake Program Teacher account;
- a real assigned program scope;
- on-track students;
- behind/needs-support students;
- missing mentor students;
- missing evidence students;
- submitted work needing review;
- revision-requested work;
- complete/ready students;
- route-backed dashboard actions;
- in-scope student detail;
- no Program Teacher access to user/security administration.

## Known Gaps

- The primary IT Program Teacher currently sees the full seeded IT program across the demo district, not a tiny classroom-only roster. The selected-site Student Directory narrows the visible list for the main click-around.
- Review decisions persist through the real review endpoints, but the demo should avoid live mutation unless the presenter intentionally wants to show approval/revision behavior.
- Site Admin assignment management is not the center of this demo. Keep the Program Teacher story focused on read/review/next-action workflows.
